# 数据采集失败任务重试设计

## 背景

当前 `数据采集` 页面已经能够展示 K 线历史补齐、增量更新和数据清理的总体状态，也能显示失败任务数量与最近错误风险提示，但仍缺少两个直接影响生产运维效率的能力：

- 无法直接看到失败补齐任务的明细，只能从聚合统计和最近记录中间接判断。
- 无法针对单个失败任务执行受控重试，只能等待下一轮 scheduler 自动处理。

这会导致线上故障排查链路偏长，尤其是在某个 `symbol + period` 因网络抖动、限流或单次写入失败而进入 `failed` 后，运维人员无法在 UI 中快速验证“当前故障是否已经恢复”。

## 目标

- 在 `数据采集` 页面新增失败任务明细展示。
- 支持针对单个 `failed` 状态的历史补齐任务执行“立即同步重试一次”。
- 保持现有架构不变，不新增依赖、不调整数据库结构、不引入新的后台 worker。
- 将重试范围限制为单任务、单轮、受控请求，避免人为操作放大 Binance 或 MySQL 压力。

## 非目标

- 不提供批量重试。
- 不提供对 `pending`、`running`、`completed` 任务的手动控制。
- 不新增暂停、取消、恢复全部任务等后台任务管理能力。
- 不修改现有 scheduler 的整体调度模型。

## 方案对比

### 方案 A：同步重试单个失败任务（推荐）

新增一个写接口，接收单个失败任务的 `symbol + period`，在接口请求生命周期内立即执行一小轮补齐逻辑，然后返回执行结果。

优点：

- 用户点击后立刻获得结果，运维反馈最快。
- 可以最大化复用现有补齐任务和补齐执行逻辑。
- 改动集中在现有 `market` 路由、`store` 聚合逻辑和 `数据采集` 页。

代价：

- 请求耗时会高于普通只读接口。
- 必须严格做互斥控制和任务状态校验，避免与 scheduler、清理任务并发写入。

### 方案 B：重置为 pending 后等待 scheduler

优点是接口轻量，逻辑简单；缺点是用户感知不是“立即重试”，且调度触发时机不稳定，不满足本次目标。

### 方案 C：独立手动重试 worker

可扩展性更高，但属于后台任务架构升级，超出当前“小改、生产优先、安全优先”的范围。

## 推荐方案

采用方案 A：新增同步重试接口，并将执行范围限制为“单个失败任务的一小轮补齐”。接口只做一次受控拉取与写入，不把手动操作升级为长期运行任务。

## 架构设计

### 1. 后端读接口增强

扩展 `GET /api/market/kline-status` 的响应模型，在现有聚合结果中新增失败任务明细列表。每个失败任务至少包含：

- `symbol`
- `period`
- `lastError`
- `updatedAt`
- `pagesFetched`
- `storedCandles`
- `nextStart`

这样前端无需新增第二个只读接口，就能直接渲染失败任务区域并执行单项操作。

### 2. 后端重试接口

新增写接口，例如：

`POST /api/market/kline-backfill/retry`

请求体只允许包含单个任务定位信息：

- `symbol`
- `period`

接口执行流程：

1. 查找对应补齐任务。
2. 校验任务存在且当前状态必须为 `failed`。
3. 检查当前是否存在不允许的互斥条件：
   - 全局 K 线清理正在执行。
   - 目标任务已被 scheduler 或其他请求抢先接管为 `running`。
4. 将目标任务短暂切换到受控执行状态，清空过期错误信息。
5. 复用现有补齐执行逻辑，仅对该任务执行一小轮补齐。
6. 根据结果更新状态并返回明确结果。

### 3. 补齐执行边界

为了防止手动重试变成“前台触发的长事务任务”，本次同步重试遵循以下限制：

- 一次请求只处理一个 `symbol + period`。
- 一次请求只执行单轮受控补齐，不循环跑完整窗口。
- 复用现有补齐页大小与分页逻辑，不引入新的大参数。
- 若外部请求再次失败，保留任务为 `failed` 并更新 `lastError`。
- 若本轮成功但未完全补齐，则让任务回到可继续推进的状态，由现有 scheduler 后续接管。
- 若本轮已补齐窗口，则更新为 `completed`。

### 4. 前端页面增强

保持 `数据采集` 页现有整体布局不变，在底部区域新增“失败任务”表格，展示：

- 交易对
- 周期
- 最近错误
- 最近更新时间
- 已抓页数
- 已写入 K 线数
- 下次起点
- 操作按钮

支持最小筛选能力：

- 按周期筛选
- 默认展示全部失败任务

交互规则：

- 点击“立即重试”后，仅锁定当前行按钮，避免整页卡死。
- 成功后刷新 `kline-status` 数据。
- 失败时展示中文错误信息，不吞错。
- 如果没有失败任务，显示明确空状态。

## 数据流

### 页面加载

1. 前端调用 `GET /api/market/kline-status`。
2. 后端返回总体卡片、周期进度、运行中任务、最近任务、风险提示和失败任务明细。
3. 前端渲染失败任务表和筛选结果。

### 手动重试

1. 用户在失败任务表点击“立即重试”。
2. 前端发起 `POST /api/market/kline-backfill/retry`。
3. 后端完成校验、互斥检查、单轮补齐执行与状态更新。
4. 前端收到结果后提示成功或失败。
5. 前端重新刷新 `GET /api/market/kline-status`，更新页面状态。

## 错误处理

需要明确区分以下错误场景，并返回可直接展示的中文错误：

- 任务不存在。
- 任务不是 `failed`，禁止手动重试。
- 当前正在清理 K 线，暂不允许重试。
- 任务已被其他执行流接管。
- Binance 请求失败或超时。
- 写入数据库失败。

错误处理原则：

- 不吞掉底层错误。
- 不返回模糊的“操作失败”。
- 前端优先展示后端返回的明确原因。

## 安全与生产约束

- 只允许已登录用户访问，沿用现有认证链路。
- 不新增绕过权限的快捷通道。
- 不修改 `.env`、密钥配置或部署配置。
- 不允许前端传入页大小、循环次数等会影响资源消耗的参数。
- 必须复用现有互斥锁和状态控制，避免与补齐 scheduler、增量采集、清理任务形成写冲突。
- 失败任务重试接口必须是幂等可理解的：同一时刻重复点击时，后到请求应被明确拒绝，而不是重复写入。

## 测试设计

本次实现采用 TDD，先补失败测试，再写最小实现。核心测试覆盖：

### 后端

- `kline-status` 能返回失败任务明细。
- 只有 `failed` 任务允许调用重试接口。
- 重试成功时返回成功结果，并正确更新任务状态与计数。
- 重试过程中若遇到清理互斥或任务已被接管，会返回明确错误。
- 重试失败时保留 `failed` 状态并刷新 `lastError`。

### 前端

- `数据采集` 页在存在失败任务时能渲染失败任务表。
- 周期筛选能正确过滤失败任务。
- 点击“立即重试”会调用对应 API，并在完成后刷新状态。

## 风险与缓解

- 风险：同步重试接口响应时间偏长。
  缓解：严格限制为单任务、单轮执行，不放开批量和无限循环。

- 风险：与 scheduler 或清理任务并发写入。
  缓解：复用现有锁与状态检查，冲突时直接拒绝请求。

- 风险：用户频繁点击导致重复请求。
  缓解：前端锁定单行按钮，后端对已被接管任务直接拒绝。

- 风险：失败原因过长影响页面可读性。
  缓解：表格中展示主要错误文本，必要时允许前端做省略展示但保留完整 title。

## 验收标准

- `数据采集` 页可以直接看到失败补齐任务明细。
- 失败任务可以逐条触发“立即重试”。
- 成功、失败、互斥冲突都能给出明确中文反馈。
- 后端与前端自动化验证通过。
- `PROJECT_PROGRESS.md` 更新本次计划、修改记录、验证结果、下一步任务与风险点。
